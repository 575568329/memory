# 记忆系统最佳实践指南

> 如何高效使用自适应记忆层

---

## 📖 使用原则

### 1. 质量优于数量

**✅ 好的记忆**：
```markdown
- 用户偏好严格遵循TDD流程：红→绿→重构
- 证据：见于2026-03-20, 03-23, 03-25的对话
- 置信度：85%
```

**❌ 差的记忆**：
```markdown
- 用户写代码
```

### 2. 分层存储

```
MEMORY.md (核心，<200行)
    ↓
分类文件 (详细)
    ↓
归档存储 (历史)
```

### 3. 定期维护

- **每周**：查看周总结，确认记忆准确性
- **每月**：归档低分记忆（<4分）
- **每季度**：清理重复或过时记忆

## 🎯 不同场景的使用方法

### 场景1：编码偏好

**何时记录**：
- 用户明确说"我喜欢..."、"我通常..."
- 同一偏好出现3次以上

**如何记录**：
```markdown
### 编码风格 - TypeScript

**偏好描述**：
- 严格模式：`strict: true`
- 禁止 `any` 类型，不得已用 `unknown`
- 接口优先于类型别名

**证据记录**：
- 2026-03-20：明确提到"不用any"
- 2026-03-23：项目中选择strict模式
- 2026-03-25：重构时移除所有any

**评分**：9.0/10（高频 + 高一致性）
```

**存储位置**：
- 评分≥7：`MEMORY.md` + `memory/preferences/coding-style.md`
- 评分4-6：`memory/preferences/coding-style.md`
- 评分<4：归档或删除

### 场景2：技术决策

**何时记录**：
- 技术选型（React vs Vue）
- 工具选择（pnpm vs npm）
- 架构决策（微服务 vs 单体）

**如何记录**：
```markdown
# 技术决策：采用pnpm作为包管理器

**决策时间**：2026-03-26

**选项对比**：
- npm：安装速度慢，依赖重复
- pnpm：安装速度快，节省磁盘空间
- Yarn：配置复杂

**选择**：pnpm

**原因**：
1. 安装速度提升2-3倍
2. 节省50%磁盘空间
3. 与npm兼容性好

**影响范围**：
- 所有新项目
- 现有项目逐步迁移

**相关决策**：
- 2026-02-01：选择TypeScript
- 2026-01-15：采用ESLint+Prettier
```

**存储位置**：
- 文件名：`memory/decisions/20260326 adopting_pnpm.md`
- 评分：重要决策≥8分
- 更新：如有变化，创建新决策记录

### 场景3：工作流程

**何时记录**：
- TDD、代码审查流程
- 调试和问题解决方法
- 项目启动和部署流程

**如何记录**：
```markdown
### TDD工作流程

**触发条件**：所有新功能开发

**流程**：
```
1. 编写失败测试（Red）
   └─ 定义接口和行为

2. 实现最小代码（Green）
   └─ 仅实现测试要求的功能

3. 重构优化（Refactor）
   └─ 改进代码质量，保持测试全绿
```

**工具**：
- 测试框架：pytest / JUnit / Vitest
- Mock工具：mockito / unittest.mock
- 覆盖率：coverage.py / JaCoCo

**例外情况**：
- 快速原型：可先实现后补充测试
- 紧急修复：先修复，后续补充测试
```

**存储位置**：
- 主流程：`MEMORY.md`（简述）
- 详细流程：`memory/preferences/workflow.md`

### 场景4：调试经验

**何时记录**：
- 重复遇到的问题
- 复杂问题的解决方法
- 性能优化经验

**如何记录**：
```markdown
### 问题：N+1查询导致性能问题

**症状**：
- 页面加载缓慢（>5秒）
- 数据库查询次数过多

**原因**：
- 循环中执行数据库查询
- 未使用批量查询

**解决方案**：
```python
# ❌ 错误：N+1查询
for user in users:
    orders = get_orders(user.id)  # N次查询

# ✅ 正确：批量查询
user_ids = [u.id for u in users]
orders = get_orders_by_ids(user_ids)  # 1次查询
```

**预防措施**：
- 使用ORM的预加载功能
- 查看日志中的查询次数
- 性能测试覆盖

**相关工具**：
- Django Debug Toolbar
- SQLAlchemy日志
- EXPLAIN查询计划
```

**存储位置**：
- `memory/debugging.md`
- 按语言/框架分类

- 项目特定配置

**如何记录**：

**技术栈**：
- 后端：Python + FastAPI

- 消息队列：RabbitMQ

**关键配置**：
```env

**业务规则**：
- 订单状态流转：待支付→已支付→发货中→已完成
- 库存扣减：下单时预占，支付时扣减
- 退款规则：7天无理由，15天质量问题

**部署流程**：
1. 运行测试：`pytest`
2. 构建镜像：`docker build`
3. 推送仓库：`docker push`
4. 更新服务：`kubectl rollout`
```

**存储位置**：
- `memory/projects/ecommerce-api.md`

## 🔄 记忆生命周期管理

### 新记忆创建

1. **检测阶段**
   - AI自动扫描对话
   - 用户显式要求

2. **验证阶段**
   - 检查是否已存在
   - 评估重要性（1-10分）

3. **确认阶段**
   - 展示提取建议
   - 用户确认或修改

4. **存储阶段**
   - 写入对应文件
   - 更新评分记录
   - Git提交

### 记忆更新

**何时更新**：
- 偏好发生变化
- 技术栈升级
- 工作流程改进

**如何更新**：
```markdown

## 📊 评分指南

### 评分标准

| 分数范围 | 存储位置 | 更新频率 | 示例 |
|---------|---------|----------|------|
| **9-10分** | MEMORY.md核心 | 每月审查 | 编程语言、核心工作流程 |
| **7-8分** | MEMORY.md + 分类 | 每月审查 | 技术栈选择、主要工具 |
| **5-6分** | 分类文件 | 每季审查 | 项目特定配置、偶尔使用的工具 |
| **3-4分** | 分类文件 | 每半年审查 | 临时项目、一次性决策 |
| **1-2分** | 归档或删除 | - | 过时信息、错误记录 |

### 评分算法

```python
score = (
    frequency * 0.3 +      # 访问频率（带衰减）
    recency * 0.3 +        # 时效性（新近度高）
    user_rating * 0.3 +    # 用户评分（1-5星）
    context * 0.1          # 上下文相关度
)
```

### 提升评分的方法

1. **增加访问频率**
   - 在相关场景中引用
   - 定期查看和验证

2. **保持新鲜度**
   - 及时更新变化
   - 记录使用情况

3. **用户反馈**
   - 高星标记重要记忆
   - 低星标记可以归档

## ⚠️ 常见错误

### 错误1：记录过于宽泛

❌ **错误**：
```markdown
- 用户会编程
- 用户使用电脑
```

✅ **正确**：
```markdown
- 用户偏好TypeScript，严格模式
- 使用Windows 10 + Git Bash开发
```

### 错误2：缺少证据

❌ **错误**：
```markdown
- 用户喜欢TDD
```

✅ **正确**：
```markdown
- 用户喜欢TDD
- 证据：2026-03-20, 03-23, 03-25三次对话中明确提及
- 置信度：85%
```

### 错误3：不及时更新

❌ **错误**：
```markdown
- 用户使用JavaScript（2025年记录，现在已改用TypeScript）
```

✅ **正确**：
```markdown
## 技术栈演进
### 2026-03-26：迁移到TypeScript
- 原因：类型安全需求
- 状态：3/5项目完成

### 2025-12-01：使用JavaScript
- 已废弃，参见TypeScript迁移
```

### 错误4：忽略上下文

❌ **错误**：
```markdown
- 使用pnpm
```

✅ **正确**：
```markdown
- 新项目使用pnpm，现有项目逐步迁移
- 例外：与CI/CD环境兼容性问题时暂缓
```
